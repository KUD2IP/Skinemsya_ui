# Технологический стек

Подбор стека подчинён четырём целям проекта: **максимальная скорость разработки**, **красивый и уникальный вид**, **простота поддержки**, **хорошая интеграция с Telegram Mini Apps**. Отдельная сквозная задача — чтобы интерфейс **не выглядел как типовая ИИ-генерация**, поэтому осознанно избегаем самых «дефолтных» для нейросетей решений.

---

## 1. Краткая сводка

| Слой | Выбор | Почему |

## 2. Почему именно так (и почему НЕ иначе)

### 2.1 React + Vite, а не Next.js

Mini App — это SPA внутри Telegram WebView. SSR/серверные роуты Next.js здесь не нужны и лишь усложняют деплой. Vite даёт самый быстрый dev-цикл и минимальный, предсказуемый бандл. Экосистема Telegram Mini Apps (SDK, примеры, шаблоны) ориентирована в первую очередь на React.

Рассмотренные альтернативы:
- **SvelteKit / SolidStart** — отличный DX, но экосистема TMA и найм под React сильнее; меньше готовых интеграций.
- **Next.js** — избыточен для SPA в WebView.

### 2.2 vanilla-extract, а НЕ Tailwind

Это ключевое решение «анти-ИИ-вида». Tailwind в разметке (`class="flex items-center gap-2 bg-slate-800 ..."`) вместе с дефолтной палитрой `slate/zinc` — самый узнаваемый признак сгенерированного нейросетью интерфейса (явно отмечено в требованиях проекта).

vanilla-extract даёт:
- **Дизайн-токены как типизированный контракт** (`theme.css.ts`) — палитра, отступы, тайминги в одном месте (см. [DESIGN_SYSTEM.md](DESIGN_SYSTEM.md)).
- **Zero-runtime**: стили компилируются в обычный CSS, нет рантайм-оверхеда (важно для скорости в WebView).
- **`recipe`** для вариантов компонентов (как у кнопки) — чисто и поддерживаемо.
- Полную уникальность визуала: ни одной «дефолтной» утилитарной палитры.

Альтернативы и почему нет:
- **Tailwind / shadcn/ui** — максимально «ИИ-дефолт», отвергнуто требованием.
- **styled-components / Emotion** — runtime-оверхед, медленнее в WebView.
- **CSS Modules** — рабочий вариант, но без типобезопасных токенов и `recipe`; vanilla-extract строже и удобнее для дизайн-системы. (Допустимо локально для мелочей.)
- **Panda CSS** — хорош, но идеологически ближе к атомарному Tailwind-подходу; выбираем более «дизайнерский» vanilla-extract.

### 2.3 Ark UI, а НЕ shadcn/Radix-по-дефолту

Нужны доступные, корректные по поведению примитивы (Dialog, Select, Switch, Tabs, Tooltip, Combobox) — но со **своим** оформлением. Ark UI (от создателей Chakra/Park UI) — headless, кросс-фреймворковый, реже выбирается ИИ, чем связка shadcn+Radix. Стилизуем его токенами vanilla-extract, оборачивая в собственные компоненты `src/shared/ui`.

Альтернатива **Radix Primitives** — тоже хороша; Ark UI выбран как более редкий и с более широким набором.

### 2.4 Motion для анимаций

`motion/react` (бывш. Framer Motion) — лучший баланс мощности и простоты: пружины, `AnimatePresence`, `layout`-анимации (для индикаторов табов), поддержка `prefers-reduced-motion`. Пресеты вынесены в `src/shared/lib/motion.ts` (см. [DESIGN_SYSTEM.md](DESIGN_SYSTEM.md), §11).

### 2.5 Phosphor Icons (duotone)

Дефолтные для ИИ наборы — Lucide и Heroicons. Phosphor с весом **duotone** даёт узнаваемый «живой» почерк и хорошо ложится на зелёную палитру (вторичный тон — акцентным цветом). Все иконки проходят через адаптер `Icon`, чтобы централизованно управлять размером/весом/цветом.

### 2.6 Rive для splash, а не «спиннер»

Запуск приложения сопровождается тематической векторной анимацией (монеты в общий котёл). Rive — лёгкий, плавный, интерактивный и редко используется — это сразу выделяет приложение. Lottie оставлен как запасной вариант (больше готовых ассетов).

### 2.7 TanStack Query + Zustand

- **TanStack Query** управляет всем серверным состоянием: кеш, инвалидация, статусы `isLoading/isError`, которые напрямую завязаны на показ скелетонов и состояний ошибки.
- **Zustand** — для небольшого клиентского состояния (сессия/токены в памяти, UI-флаги). Без редьюсеров и бойлерплейта.

### 2.8 TanStack Router

Типобезопасный роутинг с типизированными параметрами. Под небольшое приложение можно было бы взять React Router, но TanStack Router даёт лучшую типобезопасность и аккуратнее масштабируется по доменам.

### 2.9 ky для HTTP

Лёгкая обёртка над `fetch` с хуками (`beforeRequest`/`afterResponse`) — удобно вешать `Authorization`, авто-рефреш токена и нормализацию ошибок (`ApiErrorResponse`). Менее «дефолтен», чем axios, и меньше весит. Детали клиента — в [API.md](API.md).

### 2.10 Формы: react-hook-form + zod

Неконтролируемые поля = меньше ререндеров (быстро в WebView). zod-схемы переиспользуются и для типов, и для рантайм-валидации (например, форма профиля `users/me/profile`).

## 4. Принципы поддержки

- **Один источник дизайна** — токены в `src/shared/theme`. Меняем визуал централизованно.
- **Переиспользуемые компоненты — в `src/shared/ui`**, не дублируем вёрстку (см. [PROJECT_STRUCTURE.md](PROJECT_STRUCTURE.md)).
- **Только реальное API** — никакой функциональности сверх бэкенда (см. [API.md](API.md)).
- **Без лишних зависимостей** — перед добавлением библиотеки проверь, нет ли уже решения в стеке.
